Requirements

  • Realizzare una versione in Java del gioco Life di Conway, come gioco zero-player.

    1. Realizzare una versione in Java del gioco Life di Conway, come gioco zero-player. Il gioco consiste nell’introdurre una Griglia di Celle il cui stato (cella ‘viva’ o cella ‘morta’) evolve come stabilito dallle regole di ConwayLife

    2. L’utente umano deve poter:

      • specificare la configurazione iniziale della griglia del gioco

      • vedere l’evoluzione del gioco in forma opportuna (si veda Problema della vista del gioco)

      • fermare e far ripartire l’evoluzione del gioco

      • pulire (a gioco fermo) la configurazione della griglia del gioco

    Requirement analysis

    Il sistema da realizzare è una simulazione del gioco Game of Life.

    Il gioco è basato su una griglia bidimensionale composta da celle. Ogni cella può assumere due stati:

    • viva
    • morta

    La configurazione della griglia evolve nel tempo attraverso passi discreti chiamati generazioni.

    Ad ogni generazione lo stato di ogni cella viene aggiornato in base al numero di celle vive presenti nelle otto posizioni adiacenti.

    Dall'analisi dei requisiti emergono i seguenti elementi principali del sistema:

    • la griglia di gioco
    • le celle della griglia
    • le regole di evoluzione
    • il meccanismo di avanzamento delle generazioni

    Il sistema deve quindi gestire una struttura dati che rappresenti la griglia e fornire le operazioni necessarie per calcolare le generazioni successive e visualizzare lo stato corrente della simulazione.

    Problem analysis

    Una prima analisi del problema si può trovare in Analsi del problema GofLife, mentre un primo schema organizzativo del sistema da realizzare si trova in Progetto ConwayLife26

    Lo schema rappresenta un modello del sistema, espresso utilizzando notazioni grafiche in stile UML. Lo riportiamo qui sotto, come prodotto della analisi del problema, prima ancora che di progetto.

    Infatti:

    1. I requisiti menzionano una Griglia di Celle. Questi termini sono formalizzati nelle classi Grid e Cell ritenendo adeguato che questi enti siano realizzati in software come POJO.

    2. Le regole di Conway sono il ‘core business’ applicativo e fanno riferimento a Grid. La specifica di queste regole è affidata a un altro POJO, istanza di una classe Life.

    3. Il gico è zero-player, ma deve essere controllato dall’utente attraverso dispostivi di I/O. Come analisti, non riteniamo che questo compito sia responsabilità di nessuna delle classi precedenti, per cui si ritiene necessario introdurre un nuovo POJO, di classe LifeController

    4. I dispositivi di I/O potrebbero essere i più diversi. Occorre rendere LifeController indipendente dai dettagli tecnologici dell’I/O definiìendo opportuni contratti (formalizzati come interface Java) che i dispostivi dovrenno rispettare.

    Test plans

    L'obiettivo del piano di testing è verificare che il sistema soddisfi i requisiti funzionali del Game of Life. I test vengono definiti già nella fase di analisi per rendere verificabili gli effetti del sistema.

    I principali test funzionali previsti sono i seguenti.

    • Test di inizializzazione della griglia
      Dato uno stato iniziale della griglia, il sistema deve rappresentarlo correttamente.
    • Test delle regole di evoluzione
      Verificare che l'evoluzione delle celle rispetti le regole del Game of Life.
    • Test della morte per isolamento
      Una cella viva con meno di due vicini vivi deve diventare morta nella generazione successiva.
    • Test della sopravvivenza
      Una cella viva con due o tre vicini vivi deve rimanere viva.
    • Test della morte per sovrappopolazione
      Una cella viva con più di tre vicini vivi deve morire.
    • Test della nascita di una cella
      Una cella morta con esattamente tre vicini vivi deve diventare viva.
    • Test di evoluzione della griglia
      Data una configurazione iniziale nota, il sistema deve produrre la generazione successiva attesa.

    I test saranno inizialmente eseguiti come test funzionali e successivamente automatizzati tramite test di unità durante la fase di implementazione.

    Project

    Il progetto viene sviluppato utilizzando il linguaggio Java e il sistema di build Gradle. Il codice sorgente è organizzato secondo la struttura standard dei progetti Java.

    Il progetto è organizzato nei seguenti componenti principali.

    • Grid
      Rappresenta la griglia di gioco e gestisce lo stato delle celle.
    • Cell
      Rappresenta una singola cella della griglia e il suo stato (viva o morta).
    • GameOfLife
      Contiene la logica di evoluzione del sistema e implementa le regole del gioco.
    • Main
      Gestisce l'avvio dell'applicazione e l'esecuzione della simulazione.

    Il codice sarà organizzato nel package conway all'interno del source folder src/main/java.

    Durante lo sviluppo verranno introdotti test automatici per verificare il corretto funzionamento dei moduli principali del sistema.
